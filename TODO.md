# Datalazo Architecture Roadmap: Multi-Client Customization & Agreements

This document outlines the strategic roadmap for scaling the Datalazo agency application to multiple customers using a **Single Codebase / Multi-Subdomain (Option B)** architecture, along with terms agreement integration.

---

## 1. Terms & Conditions Agreement Integration (Clickwrap Flow)
Ensure customers agree to your legal terms upon logging into their custom dashboard.

- [ ] **Database Schema Update**
  - [ ] Add `termsAccepted` (Boolean, default `false`) and `termsAcceptedAt` (DateTime, optional) fields to the User/Client model in Prisma.
  - [ ] Run Prisma migration to sync the database.

- [ ] **Dashboard Check & Clickwrap Modal**
  - [ ] Create a modal component that overlays the dashboard if `termsAccepted` is `false`.
  - [ ] Disable closing of the modal (no "X" button, no background clicking).
  - [ ] Add a scrollable window containing the Terms & Conditions text.
  - [ ] Include an *"I agree to the Terms & Conditions and Service Agreement"* checkbox.
  - [ ] Bind the "Process / Enter Dashboard" submit button to the checkbox state (disabled until checked).

- [ ] **Save Agreement Metadata**
  - [ ] Create an API endpoint (`POST /api/user/accept-terms`) that updates `termsAccepted` to `true`, records the exact timestamp, and logs the user's IP address/user-agent for legal compliance audit trails.

---

## 2. Stripe Billing Setup (Admin-Controlled)
Keep the setup simple for customers by hosting the payment relationship on the Admin's side.

- [ ] **Admin Stripe Integration**
  - [ ] Admin sets up the master Stripe account or sub-accounts.
  - [ ] Place the Admin's Stripe keys (`STRIPE_SECRET_KEY` and public keys) inside the subdomain's local `.env` file.
  - [ ] Customers pay standard invoices or subscription checkout URLs generated by the system without needing their own Stripe accounts.

---

## 3. Option B: Multi-Subdomain Deployment Strategy
Deploying dedicated white-labeled portal instances (e.g., `clientA.datalazo.net`) pointing to one core Git repository.

- [ ] **Single Repository, Multiple Targets**
  - [ ] Link all subdomain hostings (e.g. Vercel, VPS) to the main Git branch for simultaneous updates.
  - [ ] Ensure `datalazo.config.json` and `.env` are in `.gitignore` to avoid environment sharing.

---

## 4. Configuration-Driven Customizations
Custom account codes and custom CSV layouts dynamically loaded per subdomain.

- [ ] **Define Schema in `datalazo.config.json`**
  - Add fields for custom accounting rules and export layout columns.
  - *Example Client Configuration:*
    ```json
    {
      "agencyName": "Client A",
      "accounting": {
        "depositAccountCode": "1010",
        "withdrawalAccountCode": "2020",
        "typeCodeDefault": "GJ"
      },
      "csvExport": {
        "columns": ["Type", "Date", "Account", "Debit", "Credit", "Description"]
      }
    }
    ```

- [ ] **Read Config Dynamic Parameters in Code**
  - [ ] Fetch the custom account codes using `getDatalazoConfig()` rather than hardcoding `'260'` and `'500'`.
  - [ ] Render the CSV download order dynamically based on the arrays defined in `csvExport.columns`.
